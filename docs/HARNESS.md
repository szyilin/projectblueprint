# Harness 协作模型

## 两个角色

| 角色 | 职责 |
|------|------|
| **人类** | 提供意图、做取舍、批准契约、验收里程碑 |
| **AI** | 按流程提问、解释权衡、起草文档、执行已批准的脚手架 |

## 信息流向

```text
人类意图（口语、模糊）
    → AI 结构化提问（INTAKE）
    → 产品契约（CONTRACT.md）
    → 架构摘要（ARCHITECTURE.md）
    → 人类批准
    → 脚手架 + 第一个里程碑
    → 人类验收（跑验证命令）
```

## 批准点（Hard Gates）

以下节点**必须**获得人类明确同意（「可以」「按这个来」「继续」等），AI 不得自行跳过：

1. **产品契约** — 做什么、不做什么、成功标准
2. **架构形态** — 单体 / 分布式 / 先单体后拆分 及理由
3. **技术栈方案** — 在 AI 给出的 2～3 个选项中择一
4. **目标目录** — 自动推断；仅冲突或扩展已有仓库时需用户确认
5. **第一个里程碑范围** — tasks 列表（建议 ≤10 条）

## 对话纪律

- 立项阶段：**一次一个问题**
- 选项题优先于开放题（降低用户认知负担）
- 每完成一个 Phase，用 3～5 句话**复述当前共识**，请用户确认
- 用户说「你看着办」时：AI 给出**推荐默认值 + 理由**，仍须用户点头发才写入契约

## 本仓库 vs 用户新项目

| | project-blueprint | 用户新项目 |
|--|-------------------|------------|
| 内容 | 流程、原则、模板 | 产品代码、契约实例 |
| AI 能否改 | 仅当用户要改进 Harness | 主要工作区 |
| 生命周期 | 长期维护的 Meta 项目 | 随业务演进 |

## 与 Harness Engineering 的关系

本仓库实践的是 **Harness Engineering** 思想：

- 重要规则写入文件（`AGENTS.md`、契约），不依赖聊天记忆
- 验证命令可执行，不靠「看起来对」
- 分阶段降低 entropy —— 先骨架后业务
