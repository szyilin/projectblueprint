# 反模式（常见弯路）

AI 引导用户从 0 搭项目时，**主动避免**以下行为。

## 流程反模式

| 反模式 | 为什么错 | 正确做法 |
|--------|----------|----------|
| 跳过产品定义直接写代码 | 需求漂移、返工 | 先 CONTRACT + 用户批准 |
| 一次生成「完整 V1」 | 不可 review、难验证 | 按 PHASES 分阶段，首里程碑 ≤10 tasks |
| 把约定只留在聊天里 | 新会话失忆 | 写入新项目 docs/ |
| 一次问 10 个问题 | 用户 overwhelm | INTAKE：一次一问 |
| 用户未确认就定技术栈 | 责任不清、后期推翻 | DECISION-GUIDE：选项 + 确认 |

## 技术反模式

| 反模式 | 为什么错 | 正确做法 |
|--------|----------|----------|
| AI 手写完整 pom.xml / 多模块树 | 易错、难维护 | Spring Initializr / 复制已验证模板 |
| 第一版就上 K8s + 5 微服务 | 运维成本远超 MVP | 默认单体或微服务就绪单体 |
| 引入 10+ 依赖「以备将来」 | 安全与升级负担 | YAGNI，契约 non-goals 约束 |
| 选冷门/停更框架 | 社区与 AI 支持差 | 推荐时注明维护状态 |
| 没有 .gitignore 就 commit | 泄漏 build 产物 | 阶段 1 必做 |

## AI 行为反模式

| 反模式 | 为什么错 | 正确做法 |
|--------|----------|----------|
| 「看起来对」就声称完成 | 幻觉 | 跑验证命令 |
| 在本 blueprint 仓库写业务代码 | 职责混淆 | 代码只写在新项目目录 |
| 替用户做所有决定 | 用户失去掌控 | 给推荐 + 等确认 |
| 复制 RelayFlow/若依 整库当模板 | scope 爆炸 | 只借鉴结构与流程 |

## 新手特有风险

| 症状 | 引导方式 |
|------|----------|
| 「我要做一个类似 XX 的系统」 | 追问：第一版只要 XX 的哪 3 个功能？ |
| 「以后可能要微服务」 | 推荐微服务就绪单体，写进 ARCHITECTURE 演进节 |
| 「我不懂数据库」 | Q10 简化选项 + 推荐一种默认（如 SQLite/PostgreSQL）并说明可换 |
| 「AI 帮我全做了」 | 强调批准点；每阶段展示摘要让用户点头 |

## Red Flags — AI 自检

出现以下念头时应 STOP：

- 「这个很简单，不用问那么多」
- 「用户赶时间，我直接选 Spring Boot」
- 「先写代码，文档后面补」
- 「一个会话搭完比较高效」
- 「blueprint 仓库里先放个 demo 也行」

**全部应回到 INTAKE / PHASES / 批准点。**
