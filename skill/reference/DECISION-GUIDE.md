# 技术选型指南（AI 用）

本文件定义 **AI 如何帮用户选技术栈**，不是「官方推荐列表」。

## 三条原则

1. **流程在本仓库，框架在对话中匹配** —— 问什么由 INTAKE 决定；用什么框架由 AI 结合用户约束 + 当前生态给出
2. **永远给选项，不给唯一答案** —— 至少 2 个可行方案 + trade-off，除非用户已有 hard constraint
3. **用户确认后才写入契约** —— 禁止 AI 擅自定栈并开始写代码

## AI 推荐时每个方案必须包含

| 字段 | 说明 |
|------|------|
| 名称 | 如「方案 A：Java 微服务就绪单体」 |
| 技术组合 | 语言、框架、DB、部署方式（各 1 行） |
| 适合理由 | 与用户的 MVP、团队、周期如何匹配 |
| 代价 | 学习成本、运维复杂度、已知限制 |
| 第一版风险 | 最可能踩坑的一点 |

## 架构形态决策树

```text
Q: 第一版需要独立扩展某一模块吗？（如 IM 要单独扩容）
├─ 否 → 倾向单体 或 微服务就绪单体
└─ 是 → 是否已有运维/K8s 能力？
         ├─ 否 → 微服务就绪单体（模块化，单部署）
         └─ 是 → 可考虑分布式

Q: 团队人数？
├─ 1～2 人 → 强烈倾向单体 / 微服务就绪单体
├─ 3～5 人 → 微服务就绪单体 或 按域拆分但单部署
└─ 5+ 且有平台组 → 可考虑分布式

Q: 第一版演示 deadline？
├─ < 2 周 → 单体 + 最少依赖
├─ 2～8 周 → 微服务就绪单体 或 标准单栈框架
└─ > 8 周 → 可按长期架构设计，仍建议分阶段交付
```

**默认推荐（无特殊约束时）：**

> **微服务就绪单体** 或 **模块化单体** —— 现在简单部署，代码边界清晰，避免过早分布式。

## 按项目类型的匹配思路（非绑定）

AI 根据下表**生成方案**，版本号以 AI 知识库为准，**写入契约时不写死 patch 版本**（除非用户要求锁定）。

| 项目类型 | 常见方向（示例，非强制） | 选型时多问 |
|----------|-------------------------|------------|
| Java 企业后端 API | Spring Boot 单体/多模块；Quarkus 若重视启动与 native | 要不要 Admin 端、多租户、工作流 |
| Python Web API | FastAPI / Django；uv 或 poetry 管理依赖 | 同步 vs 异步、ORM 偏好 |
| Python 爬虫 | Scrapy（站点多）、httpx+BS4（轻量）、Playwright（强 JS） | 规模、反爬、调度、存储 |
| Python LLM 应用 | FastAPI + LangChain/LlamaIndex/纯 SDK；向量库看规模 | RAG vs Agent、本地 vs API 模型、GPU |
| 前端 SPA | Vue/React + 组件库；Vite 构建 | 要不要 SSR、管理端路由规范 |
| 全栈 | 前后端分离 vs BFF；先定 API 契约 | 是否同仓、谁先做 |

## 何时加载 Stack Profile

确定 `stack.domain` 后，读 `docs/stacks/<domain>/`：

- 补充**该域特有问题**（见各目录 `questions.md`）
- 参考**架构分叉**（见 `decision-forks.md`）
- 参考**第一里程碑范围**（见 `first-milestone.md`）

Profile **不覆盖** DECISION-GUIDE 的三条原则。

## 禁止的推荐方式

- ❌ 「2026 年大家都用 X，就用 X」—— 无 trade-off
- ❌ 一次推荐 5 个以上方案 —— 选择瘫痪
- ❌ 推荐已停止维护的框架 —— AI 应注明维护状态
- ❌ 因 AI 熟悉某栈就默认选中 —— 须对齐用户约束与 MVP
- ❌ 第一版就推荐 K8s + 5 微服务 —— 除非用户明确要求且有能力

## 用户说「你帮我选」

AI 应：

1. 声明推荐项：`我推荐方案 B，因为 …`
2. 说明若选 A/C 会怎样
3. 问：**「是否按方案 B 写入架构文档？」**

用户同意 → 写入 `stack.selected` 并继续。
